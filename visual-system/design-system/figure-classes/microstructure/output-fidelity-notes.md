# Microstructure Output Fidelity Notes

Status: CANDIDATE class grammar, pending human Visual Art Director review. Not approved.

These notes set the output-fidelity law for the Microstructure (`footprint_dom_delta_sequence`) class. They inherit the Visual Language Specification (`VISUAL_PRODUCTION_PROTOCOL.md` section 7.1) and the output-fidelity layer (protocol section 11.5).

Writing convention: no em dashes.

Honesty boundary: everything below is design intent and target specification. No fidelity claim can be asserted as met until a real renderer exists, the gate is green, and an actual PNG and print proof are reviewed by a human. See section 9. This candidate set produced no rendered figure, no PNG, and no export, and makes no fidelity pass or fail claim about any production asset.

## 1. SVG viewBox expectations

- Fixed `viewBox`: `0 0 1280 720`. This is the full-column width (protocol section 7.1) and matches the calibration spec `export` (1280 by 720).
- The `viewBox` is the single source of coordinate truth. Intrinsic `width` and `height` may be set for raster fallback but must preserve the aspect ratio; do not letterbox or stretch.
- Coordinates use the 8 px grid. The content box is inset 32 px on all sides (x 32 to 1248, y 32 to 688).
- The SVG is self-contained: no external image references, no remote fonts fetched at render time, no scripts in the export artifact (`README.md` architecture rules; protocol section 14.6).

## 2. Print-readiness

- Textbook plates are print artifacts first (protocol section 7.1). The primary print target is vector PDF (protocol section 7.1, section 15.3).
- Vector integrity must be preserved end to end: paths, text, and strokes remain vector in the PDF, not rasterized.
- At full-column print width, hairline strokes must remain visible and all labels must remain legible. If a stroke or label cannot survive print, it must be thickened or enlarged within the token scale, or the density must be reduced.
- Dark-surface design is the default. A light print variant, if ever required for legibility, is a Design Authority decision and a separate recorded variant (protocol section 15.3), not an ad hoc change.
- Color profile policy for print is a required pre-production decision (protocol section 15.3) and is not resolved here.

## 3. PNG fallback expectations

- The raster fallback is 300 dpi PNG (protocol section 7.1, section 15.3).
- PNG is generated by `npm run snapshot:png <figure_id>` and written to `out/png/<figure_id>.png` (`README.md`). The headless browser is a visual-review rasterizer only and is off the export critical path; it is not part of the deterministic export thread.
- The PNG is for human review and for environments that cannot place vector art. It is not the source of truth; the SVG and PDF are.
- PNG generation is a later renderer task and is out of scope for this candidate set. Do not run snapshot generation as part of grammar authoring.

## 4. Safe areas

- The safe area is the 32 px outer margin. No evidence mark, label, callout, or rail text is placed outside the content box.
- Each region has its own internal padding so content does not touch region borders; the master plate uses 16 px internal padding inside panels.
- The status banner and footer sit inside the safe area, not in the margin.
- Bleed is not used; this is a placed figure, not a full-bleed page element.

## 5. Metadata expectations

- Frozen and exported assets record: asset version, spec version, class grammar version, renderer identity, generator identity if generative, and the source commit (protocol section 11.5, section 13.3).
- The figure id is stamped as a machine attribute (`data-figure-id`) for traceability, not as visible reader content.
- Approval metadata (Domain Owner sign-off, Visual Art Director sign-off, approval date) is recorded only when a real human approval exists. A renderer never writes approval metadata on its own (protocol section 13.3; `CLAUDE.md`).
- The candidate master plate in this folder carries a visible candidate status and an authoring comment block instead of approval metadata, because no approval exists.

## 6. File naming

- Committed class artifacts follow the protocol path pattern (protocol section 8.3): `design-system/figure-classes/<class>/design-brief.md`, `approved-master-plate.svg`, `layout-rules.md`, `failure-examples.md`, plus the sprint deliverables (`renderer-implementation-notes.md`, `class-specific-checklist.md`, `accessibility-notes.md`, `output-fidelity-notes.md`) per protocol section 10.2.
- The class folder is `microstructure` per task instruction; the protocol pattern literal is `<figure_type>` which would be `footprint_dom_delta_sequence`. This naming reconciliation is a human decision (see `design-brief.md`).
- Per-figure outputs follow `out/png/<figure_id>.png` and the temporary review-packet paths (protocol section 8.4). Those are temporary, must not be committed unless re-chartered, and are not produced by this task.
- The master plate filename `approved-master-plate.svg` is a protocol-required artifact name (protocol section 8.3). The filename contains the word "approved" but the file content explicitly marks itself as a candidate pending human approval. The filename is not an approval claim.

## 7. Export review

- The renderer is not done until the gate is green, the PNG is generated, the PNG is opened, the checklist has no failures, and both human sign-offs are recorded (protocol section 9 step 5, section 11.6).
- Export review confirms: correct dimensions, vector integrity, safe areas respected, correct file naming, metadata present, and that the exported artifact matches the approved composition.
- A green gate and a generated PNG are necessary, not sufficient. They do not approve the plate (protocol section 3, section 11.6).
- Export must be static and motion-free in every case (protocol section 12).

## 8. Token and encoding fidelity at export

- The exported asset must use token colors only; no hardcoded hex survives into the renderer or the export (protocol section 7.3 enforced).
- Font families declared in the SVG must be available at export so prices and labels render in the intended families; print embedding is a required pre-production test (protocol section 15.2).
- Numeric alignment must hold at export: monospace columns stay aligned in both PDF and PNG.

## 9. What cannot be claimed until a real renderer and PNG review exist

The following are explicitly NOT claimed by this candidate set and must not be inferred:

- That any Microstructure figure renders correctly on screen or in print.
- That contrast, colorblind safety, or grayscale survivability are met on a real export. These are design intent here and require verification on the actual asset.
- That the gate passes for a Microstructure figure rendered under this grammar.
- That a PNG has been generated, opened, or reviewed.
- That the master plate is approved, or that any sign-off has been recorded.
- That the legacy renderer conforms to this grammar. It does not; it carries known token drift and is a legacy candidate pending revalidation.

Output fidelity is confirmed only by reviewing the actual exported artifact at the actual output sizes, by a human, after the renderer is aligned to an approved grammar and composition. Until then, this file is a target, not a result.
